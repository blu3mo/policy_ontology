# プロジェクトの目的

このリポジトリは、**日本の国政・行政・政策・政治問題に関する高精度の構造化調査**のためのものです。

ここでいう「国政調査」とは、永田町・霞が関の公式過程だけを追うことではありません。**政策・法案・行政対応・政治的争点に影響した、周辺社会の出来事、外部アクター、海外動向、市場・技術・事故・司法・自治体の動きまで含めて追跡する**ことを意味します。

既定の成果物は、次の 4 つです。
- 時系列の再構成
- 人物・組織・制度上の関係整理
- イベント同士の関係性推定
- 公開可能な要約と可視化の下準備

## 最重要原則

1. **事実と推測を分ける。**
2. **すべての主張に出典トレースを残す。**
3. **可能なら一次資料を優先する。**
4. **不確実性は消さずに明示する。**
5. **別イベントを安易に一つへ統合しない。**
6. **人物・組織・場所・資料は複数リンクを許容する。**
7. **同じ資料が複数イベントに関係しても、イベントごとの要約は別に書く。**
8. **政治・法務・選挙・名誉毀損に関わる記述は保守的に扱う。**
9. **ユーザー向け要約は原則日本語。**
10. **断定表現は、出典が十分強いときだけ使う。**
11. **国政の外側で起きた出来事でも、政策形成や政治判断に影響するなら重要イベント候補として扱う。**
12. **政府内部の制度過程と、社会・市場・国際環境・司法・自治体側の圧力や反応を切り分けて記録する。**
13. **政府・国会・制度外要因・反応・波及の全体を見に行く。**
14. **特定テーマの既成ストーリーに合わせず、イベントの収集時点では広く拾う。**

## 必ず確認する観点

どのテーマでも、少なくとも次を確認すること。
- 何が問題として顕在化したのか
- それを可視化した統計・事故・被害・実務上の障害は何か
- 誰が論点を押し上げたか。政府、与野党、自治体、企業、団体、研究者、海外当局、市民など
- どの制度過程に乗ったか。法案、予算、通知、審議会、行政処分、判決など
- どの外部要因が制度設計に影響したか。国際情勢、価格、災害、技術進歩、市場変化、事件事故など
- どんな反発・代替案・修正要求が出たか
- 執行後にどんな影響、再修正、フォローアップがあったか

## 日本の国政調査で優先する情報源

優先順位は原則として次の通りです。

1. 官邸・内閣官房・内閣府の公式ページ
2. 各省庁・外局・庁・独立行政委員会・規制当局の公式ページ
3. 国会会議録、法案情報、委員会資料、審議会・有識者会議・検討会の議事録や配布資料
4. 法令、告示、通達、パブコメ結果、白書、年次報告、検査報告、裁判資料、公的調査報告
5. 地方自治体、国際機関、海外当局、外国議会、海外規制当局などの公的資料
6. 政党・議員本人の公式声明、記者会見、会見録、SNS 公式発信
7. 企業・プラットフォーム・業界団体・労組・NPO・学会・研究者・被害者団体の一次資料
8. 信頼できる報道機関の原資料付き報道、調査報道、ファクト付き特集
9. 解説・評論・二次分析

## 政策形成エコシステムとして見ること

調査では、次の 3 層を常に意識すること。

### 1. 公式制度過程
- 法案提出、閣議決定、答弁、通知、通達、審議会、行政処分、判決など

### 2. 制度外だが政策形成に影響する要因
- 重大事故、消費者被害、災害、サイバー事案、企業不祥事、業界慣行、海外規制強化、国際交渉、研究報告、世論喚起、キャンペーン、ロビー活動など

### 3. 制度への入力・反応・波及
- 要望書、提言、パブコメ、訴訟、行政相談、業界の自主ルール変更、プラットフォーム規約変更、自治体独自対応、国際機関勧告、株主提案、第三者委員会報告など

イベントは、上記 3 層のどこに属するかを意識し、必要なら summary や notes_uncertainty に明記する。

## 日本向けのイベント類型

イベントは原則として、次のような単位で切ること。
- 法案提出、修正、審議、成立、廃案
- 閣議決定、骨太方針、政府方針決定、対策パッケージ公表
- 省令・告示・通知・通達・ガイドライン・Q&A・制度運用変更の発出
- 審議会・検討会・有識者会議・WG・研究会の開催、資料公開、報告書公表
- 国会質疑、委員会、参考人質疑、答弁、附帯決議、会派・政党の反応
- 官邸・省庁・政党・議員による会見や声明
- 予算措置、補正、制度改正、公募、行政処分、勧告、検査、改善命令
- 裁判・仮処分・判決・上訴・和解などの司法上の節目
- 地方自治体の条例、要望、先行措置、独自調査
- 海外当局・国際機関・外国議会の公表や決定
- 企業・プラットフォーム・業界団体の対応、規約変更、自主ルール改定
- 市民団体・専門家・被害者団体・学会の提言、調査、公表、告発
- 事故・事件・被害拡大・市場変化・技術変化など、政治判断に影響した外部事象
- 指標公表、統計改定、需給逼迫、価格急変、人口動態の変化など問題認識を押し上げたデータ事象

## 探索漏れチェック

以下は見落としやすいので必ず確認する。
- 予算・税制・基金・補助金・交付金との関係
- 自治体の先行事例や実務負担
- 海外制度や国際ルールの参照
- 業界団体、職能団体、労組、当事者団体の反応
- 訴訟、仮処分、監督当局判断
- 調査報道、第三者委員会、検証報告
- 統計や白書の使われ方
- 実施後の副作用、修正、再検討

## 必須データ方針

各イベントには最低限、次を持たせること。
- `event_id`
- `topic`
- `title`
- `summary`
- `event_date_text`
- `event_date_start`
- `event_date_end`
- `timezone`
- `places`
- `people`
- `organizations`
- `source_ids`
- `event_type`
- `confidence`
- `notes_uncertainty`

推奨追加項目:
- `event_domain` 例: `公式制度過程` `社会的外部要因` `制度への入力・反応`
- `policy_stage` 例: `問題顕在化` `論点形成` `審議` `執行` `司法` `事後評価`
- `jurisdiction` 例: `日本国` `地方自治体` `海外` `国際`

各イベント間リンクには最低限、次を持たせること。
- `link_id`
- `from_event_id`
- `to_event_id`
- `relation_type`
- `rationale`
- `evidence`
- `confidence`

## 日本向けの関係タイプ初期集合

既定では以下を使う。
- `言及する`
- `応答する`
- `発端となる`
- `実施する`
- `改正する`
- `修正する`
- `調査する`
- `批判する`
- `支持する`
- `反論する`
- `制度的に引き継ぐ`
- `同一案件クラスター`
- `同一制度過程クラスター`
- `司法上の進展`
- `報道上の反応`
- `国会フォローアップ`
- `社会的圧力となる`
- `制度外要因として影響する`
- `自治体先行事例となる`
- `海外動向として参照される`
- `業界対応を促す`
- `被害・事故を受けて生じる`
- `統計・需給・価格変動を受けて生じる`

以下のどれかを満たさない限り、リンクを張らないこと。
- 片方の資料が他方のイベントを明示的に参照している
- 共通の人物・組織・制度過程があり、連続性が明確
- 日付の連鎖と制度上の流れに合理的な根拠がある
- 社会・市場・司法・海外動向が政策形成に影響したことを、1〜3 文で証拠付きで説明できる
- 1〜4 文で証拠付きの説明ができる

## 調査時の表現ルール

- 「違法」「虚偽」「腐敗」「癒着」などの強い評価語は、出典が明示的にそう述べる場合に限る。
- 疑惑、告発、申立て、報道、政府説明、司法判断、国会追及は必ず区別する。
- 日付は、可能であれば `event_date_text` と ISO 正規化の両方を保存する。
- 引用は最小限にし、発言主体と文脈を明示する。
- 既知でない情報は補完せず、`null` または不確実性メモに残す。
- 政策形成への影響を推定する場合は、必ず「何を通じて影響したか」を書く。

## テーマ別パイプライン

テーマを設定したら、以下のスキルを順番に実行して一気通貫で調査を完成させる。各ステップの出力が次のステップの入力になる。

### ステップ1: 資料の広域収集
```
/intake-source-batch {テーマ名}
```
官邸・国会・各省庁を起点に、社会・市場・司法・自治体・海外動向まで候補資料を広く集める。サブエージェントを並列起動して探索量を最大化する。

このステップは、最初の一巡で終わりにしない。一度広く拾い切った後に、収集済み資料の偏り・抜け・弱い系列を点検し、「まだ押し広げられる枝がないか」「取りこぼした周辺主体や外部要因がないか」を確認して、追加の広域収集をもう一段入れる。

**出力**: `data/{theme-slug}/processed/source_intake_*.md`, `source_links.ndjson`

### ステップ2: イベント正本の作成
```
/collect-events {テーマ名}
```
収集した資料からイベントを抽出し、正本データを作る。公式制度過程だけでなく外部要因も独立イベントとして記録する。

このステップでも、一度イベント化をやり切った後に再点検を必ず挟む。イベントの粒度、独立させるべき外部事象、欠けている転換点、薄い要約、未分離の複合イベントを見直し、「もう一段階イベントとして起こせるもの」がないかを洗ってから追加抽出を行う。

**出力**: `data/{theme-slug}/processed/events.ndjson`, `entities.ndjson`

### ステップ3: イベント間リンクの作成
```
/link-events {テーマ名}
```
9軸のサブエージェントを並列展開し、イベント間の因果・参照・制度的連続性を証拠付きで結ぶ。

**出力**: `data/{theme-slug}/processed/event_links.ndjson`

### ステップ4: QA
```
/qa-event-db {テーマ名}
```
欠損、重複、弱い根拠、参照切れ、スキーマ逸脱を点検する。問題が見つかれば前のステップに戻って修正する。

**出力**: `outputs/{theme-slug}/qa_report_*.md`

### ステップ5: 可視化の生成
```
/build-timeline-view {テーマ名}
```
`templates/policy-explorer/` の汎用テンプレートをコピーし、テーマ固有のデータ5ファイル（`config.js`, `graph.js`, `sourceLinks.js`, `network.js`, `narrative.js`）だけを生成する。テンプレートのReactコード自体は変更しない。アプリは3つのビューモードを持つ: ストーリー（スクロールテリング）、タイムライン（自由探索）、アクター（主体間ネットワーク + ピボット探索）。

**出力**: `outputs/{theme-slug}/timeline_graph.json`, `outputs/{theme-slug}/policy-explorer/`

### 追加サイクル

QA や可視化の過程で不足が見つかれば、ステップ1〜3に戻って追加探索・追加抽出・追加リンクを行う。パイプラインは一方通行ではなく、フィードバックループとして運用する。

## ツール運用方針

- 独立性が高い作業はサブエージェントへ委譲する。
- 書き込み前後は hooks と schema をガードレールとして使う。
- 不要なファイルを大量に読み込まず、現在の作業に関係あるものだけ読む。
- 大きいファイルを書き換えるより、NDJSON に追記する運用を優先する。
- テーマによっては、政府・国会以外の資料群を最初から探索対象に含める。

## ディレクトリ構造

```
├── schemas/                         # 必須フィールド定義（全テーマ共通）
├── templates/
│   └── policy-explorer/            # 可視化アプリの汎用テンプレート（全テーマ共通）
│       ├── src/
│       │   ├── App.jsx             # 3モード管理（ストーリー/タイムライン/アクター）
│       │   ├── TimelineViz.jsx
│       │   ├── DetailPanel.jsx     # 資料リンク + アクターへの遷移
│       │   ├── OrgNetwork.jsx      # アクターネットワーク（D3 force + ピボット探索）
│       │   ├── index.css
│       │   ├── main.jsx
│       │   └── data/
│       │       └── layout.js       # config.js から年範囲を読む
│       ├── index.html
│       ├── package.json
│       └── vite.config.js
├── data/
│   └── {theme-slug}/               # テーマ別データ（例: choson-gikai/）
│       └── processed/
│           ├── events.ndjson
│           ├── event_links.ndjson
│           ├── entities.ndjson
│           ├── source_links.ndjson
│           └── source_intake_*.md
├── outputs/
│   └── {theme-slug}/               # テーマ別成果物
│       ├── timeline_graph.json
│       ├── timeline_*.csv
│       ├── qa_report_*.md
│       └── policy-explorer/        # テンプレートコピー + データ5ファイル
│           └── src/data/
│               ├── config.js       # ← 生成: タイトル・年範囲
│               ├── graph.js        # ← 生成: ノード+エッジ
│               ├── sourceLinks.js  # ← 生成: 資料リンクマップ
│               ├── network.js      # ← 生成: アクターネットワーク+名前マッピング
│               └── narrative.js    # ← 生成: ストーリーステップ
└── .gitignore                       # data/*/processed/ と outputs/*/ を除外
```

`{theme-slug}` はテーマの短い英語識別子（例: `choson-gikai`, `generative-ai-regulation`, `child-support-policy`）。各スキルはこのslugを使ってデータの読み書き先を決定する。
