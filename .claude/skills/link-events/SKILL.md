## 目的
`data/{theme-slug}/processed/event_links.ndjson` に、証拠付きのイベント間関係を追記する。
サブエージェントを9軸で並列起動し、制度内外すべての接続を全探索する。

## 入力
- `data/{theme-slug}/processed/events.ndjson`
- `data/{theme-slug}/processed/source_links.ndjson`
- `data/{theme-slug}/processed/event_links.ndjson`（既存リンクの重複チェック用）
- `schemas/event_link.schema.json`

---

## フェーズ 1 ― 準備（コーディネーター自身が実行）

1. `events.ndjson` を読み込み、対象テーマのイベント全件のIDとタイトルをリストアップする。
2. `event_links.ndjson` を読み、既存リンクの `(from_event_id, to_event_id)` ペアを把握する。
3. `source_links.ndjson` を読み、各イベントに紐づく資料要約を把握する。
4. 上記3点をフェーズ2の各サブエージェントへの指示に含める。

---

## フェーズ 2 ― 9軸並列探索（サブエージェントを同時起動）

**以下の9軸を一斉に並列サブエージェントとして起動する（Agent ツール使用）。**
各サブエージェントには次を渡すこと:
- 対象テーマ
- イベント全件のID・タイトル・日付・summary 一覧
- 既存リンクペア一覧（重複除外のため）
- 担当軸の探索指示（下記）
- 出力フォーマット: JSON 配列 `[{from_event_id, to_event_id, relation_type, rationale, evidence, confidence}, ...]`

### 軸 A: 制度過程連鎖
法案提出→修正→審議→成立→施行、閣議決定→省令→通知→運用変更など、制度的に連続する行為の連鎖をすべて探す。「制度的に引き継ぐ」「実施する」「改正する」「修正する」「同一制度過程クラスター」が中心。

### 軸 B: 主体共有（人物・組織）
同一の人物または組織が複数イベントに登場し、制度的・組織的な継続行為を示すケースをすべて探す。偶然の同名ではなく、継続的役割である根拠を示す。

### 軸 C: 資料内明示参照
`source_links.ndjson` の `event_scoped_summary_ja` および `evidence_snippet` の中で、他のイベントの日付・決定内容・文書名・人名を明示的に参照しているものをすべて探す。「言及する」「応答する」が中心。

### 軸 D: 政策ライフサイクル連鎖
問題顕在化 → 論点形成 → 審議 → 執行 → 反応 → 再修正 という政策ライフサイクル上の接続をすべて探す。前後ステージの橋渡しになるイベントを重点的に拾う。「発端となる」「応答する」「国会フォローアップ」が中心。

### 軸 E: 外部要因→制度接続
自治体先行事例、司法判断、海外当局・国際機関の動向、企業・業界団体の対応、重大事故・被害統計・市場変化が制度過程のイベントに影響した接続をすべて探す。「社会的圧力となる」「制度外要因として影響する」「自治体先行事例となる」「海外動向として参照される」「業界対応を促す」「被害・事故を受けて生じる」が中心。

### 軸 F: 国会フォローアップ
国会質疑・委員会答弁・附帯決議・参考人意見が、行政側の措置や法案修正・通知発出・次の質疑とどう連鎖しているかをすべて探す。「国会フォローアップ」「批判する」「支持する」「反論する」が中心。

### 軸 G: 予算・統計・指標連鎖
予算措置・補正・交付金、統計公表・白書、需給変化・価格変動・人口動態・被害件数増減が、政策判断イベントに影響した接続をすべて探す。「統計・需給・価格変動を受けて生じる」「発端となる」が中心。

### 軸 H: 司法・行政処分連鎖
提訴→仮処分→判決→上訴→和解、または行政調査→処分→不服申立→再処分などの司法・行政的連鎖をすべて探す。「司法上の進展」「調査する」「制度的に引き継ぐ」が中心。

### 軸 I: 逆方向・見落とし補完検証
軸 A〜H で発見されたリンク候補を逆方向から読み直し、逆向きリンクが成立するかを検証する。また、孤立している（リンクが0件の）イベントを特定し、それらと他イベントの間に見落とされた接続がないかを重点的に探す。既存 event_links.ndjson のうち `confidence: low` のものも再検証する。

---

## フェーズ 3 ― 統合・重複除去・品質フィルタ（コーディネーター）

1. 全サブエージェントの結果を収集する。
2. `(from_event_id, to_event_id)` が同一の候補が複数ある場合、最も証拠が強いもの（evidence スニペットが具体的なもの）を採用し、残りは棄却記録に残す。
3. 既存リンクペアと重複するものを除外する。
4. 以下の基準でさらに却下する:
   - rationale が「〜と思われる」「〜の可能性がある」のみで、具体的事実を含まない
   - evidence が資料引用なしで「〜から推測される」のみ
   - 話題の近接性だけが根拠（共有主体・資料参照・手続連鎖がいずれもない）
   - 裏の意図や非公開調整の推測
5. 通過した候補に `link_id`（`lnk_YYYYMMDD_NNN` 形式、NNNは3桁連番）を採番する。

---

## フェーズ 4 ― 書き込み

`data/{theme-slug}/processed/event_links.ndjson` に NDJSON 形式で追記する（1行1レコード）。

---

## 各リンクの必須項目
- `link_id`
- `from_event_id`
- `to_event_id`
- `relation_type`
- `rationale`
- `evidence`
- `confidence`

## relation_type の選択肢（CLAUDE.md 準拠）
`言及する` / `応答する` / `発端となる` / `実施する` / `改正する` / `修正する` / `調査する` / `批判する` / `支持する` / `反論する` / `制度的に引き継ぐ` / `同一案件クラスター` / `同一制度過程クラスター` / `司法上の進展` / `報道上の反応` / `国会フォローアップ` / `社会的圧力となる` / `制度外要因として影響する` / `自治体先行事例となる` / `海外動向として参照される` / `業界対応を促す` / `被害・事故を受けて生じる` / `統計・需給・価格変動を受けて生じる`

## リンクを張れる条件（いずれか一つ以上必須）
- 片方の資料が他方のイベントを明示的に参照している
- 共通の人物・組織・制度過程があり、連続性が明確
- 日付の連鎖と制度上の流れに合理的な根拠がある
- 社会・市場・司法・海外動向が政策形成に影響したことを 1〜3 文で証拠付きで説明できる

## 禁止
- 話題が近いだけでリンクしない
- 裏の意図や非公開調整を推定しない
- 弱い根拠で high confidence を付けない

---

## 完了時に返すこと
- 軸別の候補数と採用数の内訳
- 採用リンク総数（confidence 別: high / medium / low）
- 却下理由の分布
- 孤立イベント（リンクが依然0件のもの）の一覧
- 追加収集が必要な箇所
- 主要な接続パターン（上位5件程度）
